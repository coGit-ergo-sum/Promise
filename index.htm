Intro

The Promise Object


The Promise object is designed for managing asynchronous processes.

For a single process, a Promise is just an option, as callbacks suffice. in all the other cases, makes the difference!

intro: Main elements.

The elements of a Promise that make it function are the constructor and the methods: 'then', 'catch', 'finally'. 

	
	var promise = new Promise(executor);

	// ATTENTION Splitting the configuration into two parts is uncommon
	//           but intentional. It makes evident that the 'onResolve'
	//           and 'onReject' callbacks can be associated with the 
	//           'Promise' at any time.
	//           Refer to sections 7B and onwards for more details.
	promise
		.then(onResolve, onReject)
		.catch(onCatch)	
		.finally(onFinally);

If we want an asynchronous function (such as an API call) to be managed by Promise, the asynchronous function must have a signature with two callback parameters.

ECMAScript 2015 (ES6) refers to this function as the "executor" 

	

	// This function illustrates a basic asynchronous process.
	// The random number 'alea' determines the function's behaviour.
	function executor(resolve, reject){

		// 'alea' is a random number between 1 and 3
		var alea = Math.floor(Math.random() * 3) + 1; 

		// ---------------------------------------------------------- //
		function timeout_Complete(){
			if (alea == 1) { resolve(alea);} 
			else if (alea == 2) { reject(alea); }
			
			// Simulate an error occurrence
			// This will trigger the 'catch' method, calling 'onCatch'.
			else{ throw new Error(alea); }	
		}
		// ---------------------------------------------------------- //


		// Waits 1 second before running 'timeout_Complete'.
		setTimeout(timeout_Complete, 1000); 
	}

		
	// ------------------------------------------------------- //
	// A fully configured 'Promise' also requires 4 callbacks.
	// ------------------------------------------------------- //

	function onResolve(info){
		// do something appropriate when executor succeded.
	}

	function onReject(info){
		// do something appropriate when executor rejected.
	}

	function onCatch(er){
		// Handle errors appropriately. 
	}
	 function onFinally(){
		// Execute final actions regardless of resolve, reject or error. 
	}
Intro: Promise DIY

The following is a possible DIY implementation of a Promise.

The structure is minimal and lacks the necessary checks to make it a professional tool, but (under optimal conditions) it functions correctly , similar to a standard Promise. 

	
	// A basic yet running DIY Promise for tutorial purposes.
	// Look at section 2A (in debug mode) for a live example.
	class PromiseDIY {

		onResolve = null;
		onReject  = null;
		onCatch   = null;
		onFinally = null;

		constructor(_executor) {
	
			const _resolve = (_value) => {
				this.onResolve(_value);
				this.onFinally();
			};
	
			const _reject = (_value) => {
				this.onReject(_value);
				this.onFinally();
			};
	
			try {
				_executor(_resolve, _reject);
			} 
			catch (_error) {
				this.onCatch(_error);
				this.onFinally();
			}
		}
	
		then(_onResolve, _onReject) {
			this.onResolve = _onResolve;
			this.onReject = _onReject;

			// implements the chaining
			return this; 
		}
	
		catch(_onCatch) {
			this.onCatch = _onCatch;

			// implements the chaining
			return this;
		}
	
		finally(_onFinally) {
			this.onFinally = _onFinally;

			// implements the chaining
			return this;
		}
	}

If the process is valid, (such as correct login data and user authentication) the 'resolve' callback is called (optionally with any 'resolve' data). Conversely, if the login data is invalid, the 'reject' callback can be called (optionally with any 'reject' data).

ATTENTION: A 'rejection' is not the same as a 'failure'.

Calling one callback or the other (within the executor function) results in the Promise calling the corresponding function with the 'then' method: either onResolve or onReject.

If an unhandled error occurs within the executor, Promise will call the function associated with the 'catch'' method.

Intro: asynchronous without promise.

When handling a single asynchronous operation, using a Promise is an option. If you're comfortable with callback programming, the 'executor' functions can be organized as follows, and the functionality remains the same. 


	// Promise-Free Asynchronous Function Implementation.
	function executor(){
		try{

			// ----------------------------------- //
			// do something asynchronous (or not)
			// ----------------------------------- //

			if(processIsOK){
				onResolve(info);
			}
			else{
				onReject(info);
			}
		}
		catch(er){
			onCatch(er);
		}
		finally{
			onFinally();
		}
	}

This indicates that for a single asynchronous function, using a Promise adds unnecessary complexity.

The situation radically changes when dealing with multiple asynchronous operations.

For a clearer understanding of how Promises operate in harmonising or synchronising multiple synchronous operations, two tools are introduced:
  • ProgressBar a graphical model that represents the progression of an asynchronous process. The progress bar visually reflects the state of the operation, updating its width and color based on the outcome of the asynchronous task, thereby providing immediate feedback on whether the process is progressing, has completed successfully, or has encountered an error.
  • PromiseGUI a graphical model that shows how Promises manage asynchronous operations.

Section 1A: 'ProgressBar': how it works.
The 'ProgressBar'
The progress bar serves as a model for an asynchronous process. It exposes a function named 'executor', which is designed to be used by a Promise. This function accepts two callbacks: resolve and reject, which are used to send messages back to the Promise object regarding the outcome of the asynchronous operation.
 
The 'Progress bar' implements an asynchronous process that runs for a random amount of time and ends up with one state chosen randomly between 4 internal states.

Timeout
1) Resolved 2) Rejected 3) Rejected 4) Rejected (internal error)
'Timeout' and 'internal error' are internal to the ProgressBar. Externally, both are seen as a rejection.

The 'Promise' expects only two responses: 'resolved' and 'rejected', the 'rejected' state is split into three different conditions to allow for a more granular analysis of the possible circumstances within an asynchronous process.

One example could be: a company (The 'Promise') calls you (the 'Start' function) for a mission, and gives you a smartphone (the callback 'resolve'). Your task is to buy some stuff. When done you must call back your company, notifying that you succeed (the message itself) and the price of stuffs (the data).
Once received a message ('resolve' xOr 'reject'), the 'Promise' is ready to call the methods listed at the introduction. 'PromiseGUI' is the tool used by this page to give a graphical representation of the action taken by the 'Promise'.

Section 4A:Promise.all.
Promise.all() - JavaScript | MDN
With the function 'Promise.all', the 'Master-Promise':
- resolves (is successful) only when all its 'promises' resolve.
- Rejects (fails) as soon any of the 'promises' rejects.
Note

When a 'ProgressBar' fails, only the associate 'PromiseGUI' shows the right state (become red) all the other 'PromiseGUI' remains in the pending state (dimmed).

Note

Now some, more in-depth, consideration regarding:
1) concurrency
2) threading

Starting from section 4A, a single 'Master-Promise' controls many, independent and simultaneous processes. This is always safe under javascript.

In fact, although a 'Promise' is controlling any number of processes, the javascript engine works on a single thread, so there is only one process running in a given time. Hence, when a 'start' function gets its 'CPU Time slot', it has the time to set its state and then notifying 'Promise' without worrying if another process might leave it in an inappropriate condition.

Section 6A:Single thread a demonstration.
This section tries to demonstrate what it means to be a single thread. In this demonstration:
- the horizontal line is time.
- the time between 2 vertical segments is always 50 milliseconds.
- there are two processes: the red and the green
- each process takes a random amount of time to complete
- an iteration is when both processes begin
- the scheduled interval between two iterations can be one of {100, 150, 200, 250, 300} [ms]
- the red process begins always before the green process and can last even more than (interval/2) [ms]
- the green process is always scheduled (interval/2) [ms] after the red process

Note: The canvas tag is not supported in Internet Explorer 8 and earlier versions.

The example shows that, regardless of any time schedule, operated by 'setInterval', every job (the function 'process') is executed only after the previous is completed (Only when the interval of 'setInterval' is long enough, the schedule will be respected.)
In other words, since the javascript 'engine' works on a single thread it does not allow Concurrency.

The key poit here is:
Two -or more- processes cannot run at the same time . (The red process and the green process will never overlap)

Section 7A:Promise.then Synchronous/Asynchronous.
A single "Promise" is completely indifferent whether the process it controls is synchronous or asynchronous.
Note

Configuring a 'Master-Promise' is a 3 steps process. 


	// 1) creating the array of promises where each element
	// is a 'Promise', controlling its own process:
	var promises = [ 
		new Promise(executor1),
		new Promise(executor2), 
		... ,
		new Promise(executorN)
	];

	// 2) creating the masterPromise:
	var  masterPromise = Promise.all(promises);

	// 3) Let the masterPromise doing its job.			
	masterPromise
	  .then( ... )
	  .finally( ... )
	  .catch( ... );

the fact is: they must be consecutive, but not necessarily in the same function!

The constraint is, masterPromise must be created, at least, immediately before the first resolve/reject is called.

What lesson did we get from session 7?

- 7A show that synchronous process stop the flow of code till they completed
- 7B & 7C show that 'Master-Promise' can't be created, after the synchronous processes it controls are complete.

Hence the process controlled by the 'Promise' MUST be asynchronous to be correctly managed!

To be a little bit more detailed, the 'Master-Promise' needs to be created before any of the resolve/reject are called. But, to create a 'Master-Promise', for example, using Promise.race(promises), we need to create the array of promises (the parameter of the function Promise.race)
To force the process controlled by each promise in promises to 'wait' till the 'Master-Promise' is correctly configured, we 'push' the process out of the current executing function, using setTimeout(start, 0);
